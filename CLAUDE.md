## Architecture

This is a browser extension forked from TWP (Traduzir-paginas-web) that adds bilingual (side-by-side) paragraph translation. Source lives in `src/`; `.output/chrome-mv3/` is generated by the WXT build.

### Build System

Uses [WXT](https://wxt.dev/) (Vite-based). Configuration is in `wxt.config.ts`. Chrome MV3 only.

- `src/entrypoints/` — WXT entry points (background, content scripts, HTML pages)
- `src/public/` — Static assets copied as-is to output (`_locales/`, `icons/`, `w3css/`)
- `src/lib/`, `src/background/`, `src/contentScript/`, `src/options/`, `src/popup/` — Source modules

### Script Loading Order

Content scripts are injected in this order (defined in `src/entrypoints/`):

1. `check.content.ts` — `document_start`, no allFrames: checks if content script is injected
2. `content.content.ts` — `document_end`, allFrames: main translation script (loads `lib/languages.js → config.js → platformInfo.js → i18n.js → specialRules.js → contentScript/showOriginal.js → enhance.js → pageTranslator.js`)
3. `mobile.content.ts` — `document_end`, no allFrames: mobile popup

Background scripts load via `src/entrypoints/background.ts` (WXT service worker entry).

### Key Source Files

| File | Purpose |
|---|---|
| `src/lib/config.js` | `twpConfig` singleton — global config store with all settings and their defaults. Used by content scripts and background alike. |
| `src/lib/specialRules.js` | Array of per-hostname CSS selector rules for sites needing custom targeting (Twitter/X, Reddit, HN, etc.). Users can add rules via the options page. |
| `src/contentScript/enhance.js` | DOM traversal logic — identifies translatable text nodes, wraps them, and handles the bilingual layout injection. `getPageSpecialConfig()` merges built-in rules with user rules. |
| `src/contentScript/pageTranslator.js` | Orchestrates translation: reads config, batches text nodes, calls background via `chrome.runtime.sendMessage`, and writes translated text back to the DOM. Handles dual-language styles (`none`, `underline`, `mask`, `separator`). |
| `src/background/translationService.js` | All translation engine implementations (Google, Yandex, Bing, DeepL) as classes extending `Service`. Handles batching, HTTP requests, response parsing, and caching. |
| `src/background/openaiPrompt.js` | OpenAI-compatible API client — prompt building, URL normalization, response parsing, debug logging. Exported as CommonJS for testability (`module.exports`). |
| `src/background/translationCache.js` | In-memory translation cache keyed by text + language pair. |
| `src/background/chrome_background.js` | Extension background entry — handles `chrome.runtime.onMessage` routing, context menus, tab language detection, and MimeType detection for PDF handling. |

### Translation Flow

1. Content script (`pageTranslator.js`) collects text nodes from the page (or from special-rule selectors).
2. Sends batches to the background via `chrome.runtime.sendMessage({ action: "translateTexts", ... })`.
3. Background (`chrome_background.js`) routes to `translationService.js`, which checks cache then calls the appropriate engine.
4. Translated strings return to the content script, which inserts them below the original paragraphs.

### Special Rules Schema

Each entry in `specialRules` (and user-configurable rules) can have:
```json
{
  "hostname": "example.com",        // string or array of strings
  "selectors": [".css-selector"],   // elements to translate
  "detectLanguage": true,           // per-element language detection
  "style": "underline"              // dual-language visual style override
}
```

### Testing

Tests use Node's built-in `node:test` and `node:assert`. `openaiPrompt.js` is the only background module that exports via CommonJS for direct unit testing. Other test files (`content-features.test.js`, `build.test.js`, `mv3-compat.test.js`) work by reading source files as strings and asserting on their content.
